# 更新前版本检查设计

**日期：** 2026-03-25

**目标：** 让 `xclean update` 在运行安装器前先检查是否存在更高版本，向用户输出清晰的进度信息，并在无法确定远端版本时快速失败。

## 问题

当前 `xclean update` 的行为是始终重新执行安装器。
从机制上说这没有问题，但与用户对 `update` 命令的常见预期并不一致。
用户通常会期待：

- 看到当前版本
- 知道是否存在更高版本
- 在已经是最新版本时避免无意义的重装

另外，`AGENTS.md` 中关于已发布版本的说明也已经过期，仍写着 `0.1.4`，而代码和 README 已经更新到 `0.1.7`。

## 产品方向

`xclean update` 应调整为“先检查、后安装”的流程：

1. 输出当前已安装版本。
2. 输出一行提示，表示正在检查更新。
3. 从公开 release 来源获取最新版本。
4. 比较远端版本和本地版本。
5. 如果本地已经是最新版本，则输出提示并成功退出。
6. 如果发现了更高版本，则输出该版本并继续执行安装器。
7. 如果无法确定远端版本，则输出错误并失败退出。

## 用户可见输出

输出风格应保持直接、朴素。
第一版目标文案按用户要求确定为：

- `当前版本：0.1.7`
- `正在检查更新中...`
- `当前是最新版本（0.1.7）`
- `发现新版本：0.1.8，开始安装...`

错误行为：

- 当版本查询失败时，把直接错误输出到 stderr，并非零退出
- 不允许静默回退成直接重装

由于用户明确要求了这组中文文案，本次改动中的 update 流程可以直接使用中文优先输出。
这次不额外扩展通用本地化。

## 远端版本来源

版本检查应以 GitHub Releases 作为“最新已发布版本”的事实来源。

推荐做法：

- 请求 GitHub Releases 的 `latest` endpoint，或等价的轻量级 URL
- 提取 release tag
- 把 `v0.1.7` 和 `0.1.7` 统一归一化为可比较的同一种表示

这样可以让版本检查对齐实际 release 流程，而不是对齐安装镜像的当前状态。

## 版本比较规则

比较逻辑必须语义化且可预测：

- 接受 `vX.Y.Z` 和 `X.Y.Z`
- 按数字组件比较，而不是按字符串比较
- 对归一化后相等的版本判定为“已经是最新”
- 如果远端版本字符串格式错误，则视为检查失败

这样可以避免 `0.1.10` 被错误地排在 `0.1.9` 前面这类问题。

## 架构改动

更新检查逻辑应放在 `Updater.swift` 中。

预期职责划分：

- `CLI.swift`
  - 在 `update` 命令流程中输出进度信息
  - 调用 updater 的专用远端版本解析方法
  - 决定当前是退出、安装还是报错

- `Updater.swift`
  - 获取最新远端 release 版本
  - 做版本归一化和比较
  - 当确实需要安装时，保留当前安装器执行行为

测试应当隔离网络相关逻辑，确保 updater 行为可以在不依赖真实 GitHub 访问的情况下验证。

## 文档改动

同步更新文档，让命令语义和版本说明保持一致：

- `AGENTS.md`
  - 把过期的 `0.1.4` release 引用改成当前版本

- `README.md`
  - 把 `xclean update` 的描述从“无条件重装”改成“检查后按需安装”

- `README.zh-CN.md`
  - 中文文档也做相同语义修正

## 安全与失败规则

该改动必须保留现有安装安全性：

- 保持当前安装目录保留逻辑
- 版本检查失败时，不安装任何内容
- 只有在“不需要更新”或“安装成功完成”时，才宣称命令成功

关于当前环境中 GitHub TLS 间歇性失败的已知问题仍然成立。
由于用户明确选择了“检查失败即退出”，这类瞬时 GitHub 失败现在应表现为命令失败，而不是触发重装回退。

## 测试预期

新增或更新测试，覆盖：

- 本地版本与远端版本相同
- 远端版本高于本地版本
- 远端版本带 `v` 前缀
- 远端版本格式错误时失败
- 远端查询失败时失败

在声明完成前，至少验证：

- `swift test`
- `swift run xclean --help`
- `swift run xclean --version`
